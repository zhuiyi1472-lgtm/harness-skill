# Harness Skill 中文说明

这个 skill 用来把 AI Agent、Skill 体系、工具集和执行入口，按 Harness Engineering 的思路重新组织成更稳定、可验证、可共享的工程形态。

核心目标不是让模型“更聪明”，而是让它：

- 更稳定
- 更少意外
- 更容易验证
- 更容易在多人协作中复用

## 核心原则

### 1. 确定性优于智能

生产环境优先稳定的 70 分，而不是偶尔 95 分、偶尔 20 分。

### 2. 最小惊讶原则

模型的行为要尽量符合团队预期，不能靠“这次它刚好理解对了”。

### 3. 失败优先设计

默认假设模型会误判、漏查、越权、走偏。

因此要先定义：

- 出错会出在哪
- 如何拦截
- 如何降级
- 什么时候必须停下让人审

## 先判任务类型

不要一上来就写 Skill 或 Agent 方案，先判断任务到底属于哪类：

| 类型 | 特征 | 推荐方式 |
|---|---|---|
| workflow | 输入稳定、步骤稳定、输出稳定 | 用 SOP、脚本、表单、校验器 |
| hybrid | 大部分路径稳定，少量判断节点开放 | 固定主流程，只把不确定节点交给模型 |
| agent | 多意图并发、信息不完备、路径未知 | 不写死路径，改为定义边界、证据和停机点 |

默认建议：

- 能做成 workflow，就不要做 agent
- 必须做 agent，就不要伪装成 workflow

## 五层 Skill 设计框架

### 第一层：不写死路径，定义边界

优先写这三件事：

- 触发条件
- 排斥条件
- 成功定义

不要默认写成“第一步、第二步、第三步”，除非任务天然可复现。

### 第二层：工具集最小完备

不要用更多工具去掩盖边界不清。

每个工具都要说明：

- 负责什么
- 不负责什么
- 和邻近工具如何区分

工具越重叠，模型越容易选错。

### 第三层：事实校验代替知识堆砌

与其补更多 prompt，不如补“必须查什么”。

至少要区分：

- 哪些必须查源码 / API / 实时数据
- 哪些可以从稳定架构中推断
- 哪些只能作为假设输出

### 第四层：防止系统性腐化

不要不断为个案打补丁。

如果一个 skill 总在长出例外规则，通常说明：

- 触发条件写错了
- 排斥条件不够清楚
- 工具边界有重叠

这时应该重构边界，而不是继续堆 prompt。

### 第五层：Eval 优先于修补

一个可复用 skill，必须压测这些点：

- 工具有没有选错
- 参数有没有传错
- 证据链够不够
- 遇到高风险时有没有停
- 证据不足时会不会乱补

配套样例见 [zh-cn-eval-samples.md](zh-cn-eval-samples.md)。

## 从 Hermes Agent 借什么

这个 skill 的架构抽象主要来自 `NousResearch/hermes-agent`。

建议重点借这些“模式”，不要机械照抄文件结构：

| Hermes 模式 | 可迁移结论 |
|---|---|
| Prompt 是分层装配的 | 把身份、上下文、技能、记忆、平台提示分层管理 |
| Tool 走统一注册表 | 工具元数据、schema、handler、可用性要集中管理 |
| Toolset 按场景裁剪 | 不同入口暴露不同工具集，不要全量开放 |
| Skill 用渐进披露 | 先暴露 metadata，再按需加载正文和 references |
| Memory 与 session 分离 | 长期事实和会话过程不要混存 |
| Memory 快照冻结 | 会话中尽量保持系统提示稳定，减少行为漂移和缓存抖动 |
| Cron 是独立入口 | 定时任务不是“普通对话换个时间执行”，而是另一类运行表面 |

详细子系统拆解见 [zh-cn-architecture-map.md](zh-cn-architecture-map.md)。

## 推荐输出结构

当你用这个 skill 做方案时，默认建议输出：

- 目标
- 约束
- 任务类型
- 边界定义
- 最小工具集
- 事实校验规则
- 失败退路
- 人工介入点
- Eval 样本

如果还涉及实现，再补：

- 模块拆分
- 状态模型
- 入口类型
- 验证方法

## 什么时候必须人工介入

以下情况默认要停：

- 涉及不可逆操作
- 涉及跨团队职责边界重划
- 证据不足但错误代价高
- 模型在不同业务结果之间做实质性选择

## 一句话判断标准

好的 harness / skill，不是让模型“想得更远”，而是让它：

- 少走错路
- 走错能停
- 停下时说得清楚
- 下一次还能复用
